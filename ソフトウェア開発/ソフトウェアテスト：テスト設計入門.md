# 効果的なテストのための「テスト設計」入門 ～バグを見つけ出すための思考法と技法～

# 1. はじめに：テスト設計の目的と重要性

ソフトウェア開発において、「テスト」は品質を確保するために不可欠な工程です。皆さんも、作成したプログラムが期待通りに動作するかを確認するために、さまざまなテストを行っていることでしょう。しかし、単に「テストをする」だけでは、ソフトウェアに潜む欠陥（バグ）を効率的に見つけ出し、品質を十分に保証することは難しい場合があります。

より効果的なテストを実施するためには、**「何を、どのように、どの程度テストするのか」を事前に計画し、設計する**プロセス、すなわち「**テスト設計**」が非常に重要になります。この資料では、効果的なテストを実現するためのテスト設計の考え方と基本的な技法について学んでいきます。

## 1.1 テストで何を目指すのか？：「品質保証」から「欠陥検出」へ

ソフトウェア開発においてテストが品質確保に不可欠であることは言うまでもありません。従来、テストは「仕様通りに動作することを確認し、品質を保証する」目的で行われてきました。もちろんこれは重要ですが、現代の開発、とくに複雑化するシステムにおいては、**テストの主目的を「欠陥（バグ）を発見すること」に置く**考え方がより重要視されています。

なぜなら、ソフトウェアから欠陥を完全になくすことは現実的に不可能だからです。だとすれば、テストの真の価値は「欠陥がないことの証明」ではなく、**「限られたリソースの中で、いかに効率的に、より多くの、そしてより深刻な欠陥を見つけ出せるか」** にあると考える方が実践的です。『単体テストの考え方/使い方』などでも、バグの発見に繋がらないテストはその価値が低いと指摘されています。

したがって、**テスト設計とは、ソフトウェアに潜むであろう欠陥を効果的に発見するための戦略を立て、具体的なテストケースを計画・作成する活動**と捉えることができます。この「欠陥検出」という視点を持つことが、効果的なテスト設計を行う上での重要な出発点となります。

## 1.2 「意味のあるテスト」とは何か？：バグを見つけるためのテスト設計

前のセクションで、テストの主目的を「欠陥（バグ）を発見すること」に置く考え方を紹介しました。これを踏まえると、「**意味のあるテスト**」とは、単にプログラムを実行して結果を確認するだけでなく、**「ソフトウェアに潜む可能性のある欠陥を効率的に見つけ出すことができるテスト」** であると言えます。

では、どうすれば「バグを見つける」テストを設計できるのでしょうか？闇雲にテストケースを増やしても、効率が悪く、重要な欠陥を見逃してしまう可能性があります。意味のあるテストを設計するには、以下の点を意識することが重要です。

- **リスクの高い箇所を狙う**
  すべてのコードが同じようにバグを含みやすいわけではありません。経験的に、特定の種類のコードには欠陥が潜んでいる可能性が高いことが知られています。テスト設計では、このようなリスクの高い箇所を特定し、重点的にテストする戦略が有効です。（これらのリスク箇所については、「2.1 テスト対象の分析」で詳しく見ていきます）
- **多様な観点からテストする**
  正常系の動作確認だけでなく、異常系の処理（エラーハンドリングなど）、想定外の入力、境界値、性能（レスポンスタイムなど）、セキュリティといった、さまざまな観点からテストケースを設計することで、多様な種類の欠陥を発見する確率を高めることができます。
- **具体的な期待結果を持つ**
  テストを実行した結果、何がどうなっていれば「成功」で、どうなっていれば「失敗（欠陥発見）」なのか、その期待される結果を明確に定義しておく必要があります。期待結果が曖昧なテストは、欠陥を見逃す原因となります。

これらの点を満たす「意味のあるテスト」を設計するためには、単に思いつきでテスト項目を挙げるのではなく、**計画的に進めるアプローチ**が大切です。具体的には、まずソフトウェアの仕様や「どこにバグが潜んでいそうか？」というリスクをよく考え、「どの部分や機能を重点的にチェックすべきか（**テスト観点**）」をはっきりさせます。次に、その観点から「どんな入力パターンや条件で試せば、効率よくバグを見つけられそうか（**テスト条件**）」を具体的に考えます。そして、これらの条件を効率よくテストに落とし込むためのテクニック、それが**テストケース設計技法**です。これらを上手に使うことが、効果的なテスト設計の鍵になります（後の章で詳しく見ていきましょう）。

## 1.3 テスト設計がなぜ必要か？：場当たり的なテストからの脱却

もしテスト設計を行わずに、思いつくままにテスト（アドホックテスト、場当たり的なテスト）を進めると、以下のような問題が発生しやすくなります。

- **テストの漏れ:** 重要な機能やリスクの高い箇所がテストされず、重大な欠陥が見逃される可能性があります。
- **テストの偏り:** 特定の機能や簡単なケースばかりがテストされ、複雑な条件や異常系のテストが不足しがちです。
- **テストの重複:** 同じようなテストが何度も行われ、時間と労力が無駄になる可能性があります。
- **テストの再現性の欠如:** テストの手順や期待結果が明確でないため、同じテストを再度実行したり、他の人がテストを引き継いだりするのが難しくなります。
- **テストの有効性の不明確さ:** どのような観点で、どの程度のテストが行われたのかが不明確なため、テスト活動全体の有効性を評価したり、品質レベルを判断したりするのが困難になります。

だからこそ、このようなリスクを避け、テスト活動の質と効果を最大限に高めるために、テスト設計が必要不可欠なのです。テスト設計は、テスト活動に計画性と体系性をもたらす「羅針盤」であり、以下のような理由からその実施が求められます。

- **品質を担保するために**
  テスト設計を通じて**網羅性**を意識することで、重要な欠陥の見逃しリスクを低減し、ソフトウェアが必要な品質基準を満たしていることへの確信を高めます。
- **効率的にテストするために**
  テストケースの**優先順位**付けや、テスト設計技法による**重複の排除**により、限られたリソースの中で最大限の欠陥検出効果を目指します。
- **信頼できる結果を得るために**
  テストの**手順と期待結果**を明確にすることで、**客観性**と**再現性**を確保し、誰がテストしても同じ結論に至るようにします。
- **テスト活動を管理するために**
  テスト設計に基づいてテストを進めることで、**進捗状況や品質レベル**を客観的に把握し、適切な意思決定を行うことが可能になります。

もちろん、すべてのテストを事前に完璧に設計できるわけではありません。探索的テストのように、設計を最小限にして実行しながら学ぶアプローチも重要です。しかし、基本的な機能や重要な品質特性については、上記のような目的を達成するために、しっかりとしたテスト設計を行うことが、高品質なソフトウェアを開発する上で不可欠なのです。

## 1.4 この資料で学ぶこと：効果的なテストケースを設計する考え方と技法

この「テスト設計入門」資料は、若手エンジニアの皆さんが、単にテストを実行するだけでなく、「**効果的なテストを設計する**」ための基本的な考え方と具体的な技法を身につけることを目的としています。

具体的には、以下の内容を学んでいきます。

- **テスト設計の基礎**
  テスト対象をどのように分析し、どのような観点でテスト項目を洗い出すか、その基本的な考え方を学びます。
- **テストケース設計技法**
  限られたテスト工数で、より多くの欠陥を発見するための代表的なテストケース設計技法（同値分割、境界値分析、デシジョンテーブルなど）を紹介し、その使い方を理解します。
- **テストレベルに応じた設計**
  ユニットテスト、結合テスト、E2E テストといった異なるテストレベルにおいて、それぞれどのような点に注意してテスト設計を行うべきかを学びます。
- **テスト戦略と計画の概要**
  個々のテストケース設計だけでなく、テスト活動全体をどのように計画し、進めていくかの基本的な考え方に触れます。

この資料を通じて、皆さんが「意味のあるテストとは何か？」を常に意識し、より効果的に欠陥を発見できるテストを設計するための「思考の道具箱」を手に入れることを目指します。テスト設計のスキルを身につけることは、皆さんが開発するソフトウェアの品質向上に直接貢献し、エンジニアとしての信頼性を高める上で、必ず役立つはずです。

次の章からは、テスト設計の具体的なプロセスと、その中で使われる考え方について、さらに詳しく見ていきましょう。

# 2. テスト設計の基礎：何を、どのようにテストするか？

テスト設計の重要性を理解したところで、次はその具体的なプロセス、つまり「何を考え、どのようにテストを設計していくのか？」という基礎的なステップを見ていきましょう。効果的なテスト設計は、場当たり的にテストケースを考えるのではなく、体系的なアプローチに基づいています。

## 2.1 テスト対象の分析：どこにバグが潜んでいそうか？

テスト設計の最初のステップは、「**テスト対象をよく理解し、分析すること**」です。これからテストしようとしているソフトウェア、機能、あるいはコードモジュールが、

- **何をするものなのか？（目的・機能）**
- **どのような入力があり、どのような出力（または振る舞い）が期待されるのか？（仕様）**
- **どのような内部構造になっているのか？（アーキテクチャ・実装の詳細、必要であれば）**
- **どのような技術要素（言語、フレームワーク、DB など）が使われているのか？**
- **どのような制約条件（性能要件、セキュリティ要件など）があるのか？**

といった点を把握することから始まります。これには、要求仕様書、設計書、ユーザーマニュアルなどのドキュメントを読んだり、開発者や関係者にヒアリングしたり、実際にソフトウェアを動かしてみたりすることが含まれます。

この分析を通じて、とくに**リスクの高い箇所**を特定することが重要です。『Good Code, Bad Code』などでも、エラーが発生しやすい箇所を特定し、そこを重点的にテストすることの重要性が述べられています。たとえば、以下のような箇所は一般的にリスクが高いと考えられます。

- **複雑なロジックを持つ部分:** 条件分岐が多い、計算が複雑など。
- **新規に追加・変更された部分:** 変更は新たなバグを生む可能性があります。
- **他のモジュールとのインターフェース（連携部分）:** データの受け渡しや呼び出し規約の誤解など。
- **境界条件を扱う部分:** 数値の最大・最小値、リストの先頭・末尾、null 値の扱いなど。
- **エラーハンドリング処理:** 例外処理やエラー回復ロジックは考慮漏れが発生しやすいです。
- **非機能要件に関わる部分:** パフォーマンスが要求される処理、セキュリティに関わる処理など。

テスト対象を深く理解し、リスクの高い箇所を把握することが、効果的なテスト観点を洗い出すための基礎となります。

## 2.2 テスト観点の洗い出し：どのような側面からテストするか？

テスト対象の分析ができたら、次は「**どのような側面（観点）からテストを行うか？**」を洗い出します。これを「**テスト観点**」と呼びます。テスト観点は、テスト設計における「チェックリスト」のようなものであり、テストの網羅性を確保し、テスト漏れを防ぐために役立ちます。

テスト観点は、テスト対象の特性や要求に応じて多岐にわたりますが、一般的には以下のようなカテゴリーで整理できます。

- **機能性 (Functionality)**
  - 仕様通りに正しく機能するか？（正常系）
  - 期待される結果が得られるか？
  - 入力に対して適切な出力が返されるか？
- **信頼性 (Reliability)**
  - エラー発生時に適切に処理されるか？（異常系、エラーハンドリング）
  - 不正な入力や予期せぬ操作に対して、システムがクラッシュしたり不安定になったりしないか？（堅牢性）
  - リソース（メモリ、CPU など）を適切に管理しているか？
- **効率性 (Efficiency)**
  - 要求される性能（レスポンスタイム、スループットなど）を満たしているか？
  - リソースを効率的に使用しているか？
- **保守性 (Maintainability)**
  - （主にコードレビューや静的解析の観点ですが）コードは理解しやすいか？変更しやすいか？テストしやすいか？
- **移植性 (Portability)**
  - 異なる環境（OS、ブラウザなど）でも動作するか？
- **ユーザビリティ (Usability)**
  - ユーザーにとって分かりやすく、使いやすいか？（主に E2E テストや探索的テストで評価）
- **セキュリティ (Security)**
  - 脆弱性（不正アクセス、情報漏洩など）はないか？

これらの観点をベースに、テスト対象の仕様やリスク分析の結果を踏まえ、「具体的にどのようなパターンをテストすべきか」をリストアップします。たとえば、「ユーザーログイン機能」をテストする場合、

- （機能性）正しい ID/パスワードでログインできるか？
- （信頼性）間違った ID/パスワードの場合、エラーメッセージが表示されるか？
- （信頼性）ID やパスワードが空の場合、どうなるか？
- （信頼性）ID やパスワードに特殊文字を入力した場合、どうなるか？
- （セキュリティ）パスワード入力欄はマスクされているか？
- （セキュリティ）連続してログイン失敗した場合、アカウントロックされるか？

といった具体的なテスト観点を洗い出すことができます。

## 2.3 テスト条件とテストケース：具体的なテスト内容の定義

テスト観点を洗い出したら、次はその観点に基づいて、より具体的な「**テスト条件**」と「**テストケース**」に落とし込んでいきます。

- **テスト条件 (Test Condition)**
  テスト対象が満たすべき、あるいはテストで検証すべき個々の**要素や属性、状況**を指します。テスト観点をさらにブレークダウンしたもの、と考えると分かりやすいでしょう。1 つのテスト観点から、複数のテスト条件が導き出されることがよくあります。
  たとえば、「間違った ID/パスワードの場合、エラーメッセージが表示されるか？」というテスト観点からは、
  - 「ID は正しいが、パスワードが間違っている」という条件
  - 「ID が間違っているが、パスワードは正しい」という条件
  - 「ID もパスワードも間違っている」という条件
    などが考えられます。
    また、「ID やパスワードに特殊文字を入力した場合、どうなるか？」という観点からは、
  - 「ID に SQL インジェクションを引き起こしそうな文字列を入力する」という条件
  - 「パスワードに非常に長い文字列を入力する」という条件
    などが考えられます。
    テスト条件を明確にすることで、「何をテストしなければならないか」が具体化されます。
- **テストケース (Test Case)**
  特定のテスト条件を検証するための、**具体的な入力値、実行手順、そして期待される結果**を定義したものです。テストケースは、実際にテストを実行する際の「指示書」となります。
  たとえば、「ID は正しいが、パスワードが間違っている」というテスト条件に対して、以下のようなテストケースを作成できます。

  - **テストケース ID:** TC-LOGIN-002
  - **テスト条件:** ID は正しいが、パスワードが間違っている場合にログインできないこと、およびエラーメッセージが表示されること。
  - **前提条件:** ユーザー ID "testuser" が登録済みであること。
  - **入力値:**
    - ユーザー ID: "testuser"
    - パスワード: "wrongpassword"
  - **実行手順:**
    1. ログイン画面を開く。
    2. ユーザー ID フィールドに "testuser" を入力する。
    3. パスワードフィールドに "wrongpassword" を入力する。
    4. ログインボタンをクリックする。
  - **期待される結果:**
    1. ログインに失敗し、元のログイン画面に留まること。
    2. 画面上に「ユーザー ID またはパスワードが正しくありません。」というエラーメッセージが表示されること。

  テストケースは、誰が実行しても同じ結果が得られるように、**具体的かつ明確**に記述する必要があります。入力値、手順、期待結果が曖昧だと、テストの再現性が損なわれ、結果の判定も主観的になってしまいます。

テスト観点、テスト条件、テストケースの関係をまとめると、以下のようになります。

**テスト観点（何をテストするか？） → テスト条件（観点を具体化） → テストケース（条件を検証する手順）**

このプロセスを経て、テストすべき内容が具体的な実行レベルまで落とし込まれます。

## 2.4 良いテストケースの条件：効果的、効率的、保守可能

すべてのテストケースが同じように「良い」わけではありません。効果的で、効率的で、そして将来にわたって維持しやすい「良いテストケース」を作成するためには、以下の点を意識すると良いでしょう。

- **有効性 (Effectiveness)**
  - そのテストケースは、欠陥を発見する可能性が高いか？
  - リスクの高い箇所や、バグが潜んでいそうなパターンを狙っているか？
- **効率性 (Efficiency)**
  - 他のテストケースと重複していないか？
  - 最小限の労力で、最大限の欠陥検出効果を狙えているか？
- **具体性 (Specificity)**
  - 入力値、実行手順、期待結果が明確に定義されており、誰が実行しても同じ結果になるか？
  - 期待結果が客観的に判定できるか？
- **独立性 (Independence)**
  - 他のテストケースの実行結果に依存せず、単独で実行可能か？
  - テストの実行順序に影響されないか？（とくに自動テストでは重要）
- **保守性 (Maintainability)**
  - 将来、仕様変更があった場合に、テストケースの修正が容易か？
  - テストケースの意図が分かりやすく記述されているか？
  - テストデータが管理しやすいか？
- **再現性 (Repeatability)**
  - 同じ条件で何度実行しても、同じ結果が得られるか？（実行環境や時刻に依存しないなど）

これらの条件を満たすテストケースを設計することは、テスト活動全体の質を高める上で非常に重要です。次の章で紹介するテストケース設計技法は、これらの条件、とくに有効性と効率性を高めるための具体的な手段を提供してくれます。

# 3. テストケース設計技法：網羅性と効率を高める武器

前の章では、テスト観点からテスト条件、そして具体的なテストケースへと落とし込んでいくプロセスを学びました。しかし、テストすべき条件は無数に存在することが多く、すべての組み合わせをテストするのは現実的ではありません。

そこで役立つのが「**テストケース設計技法**」です。これは、**より少ないテストケースで、より多くの欠陥を効率的に発見する**ために、先人たちが編み出してきた経験的なテクニックや考え方のことです。テストケース設計技法を適用することで、勘や経験だけに頼るのではなく、**体系的かつ網羅的にテストケースを導き出す**ことが可能になります。

この章では、代表的なテストケース設計技法をいくつか紹介します。

## 3.1 ブラックボックステスト技法

ブラックボックステスト技法とは、ソフトウェアの**内部構造（コードがどう書かれているか）には着目せず**、**入力と出力（仕様）** に基づいてテストケースを設計する技法群です。まるで中身が見えない「黒い箱」を扱うように、どのような入力を与えたら、どのような出力が返ってくるべきか、という観点からテストを考えます。

この技法は、要件定義や仕様書からテストケースを作成する場合や、システムの外部からの振る舞いを検証する際にとくに有効です。

### 3.1.1 同値分割法：同じ扱いのデータをグループ化する

- **どんな考え方か？**
  入力データや出力データの中で、「**同じように扱われるであろうデータの範囲（グループ）**」を見つけ出し、各グループから**代表的な値を 1 つ**選んでテストケースとする、という考え方です。もし、あるグループ内の代表値でテストが成功（または失敗）すれば、同じグループ内の他の値でも同様の結果になるだろう、と推測することで、テストケースの数を大幅に削減します。この「同じように扱われるデータのグループ」を「**同値クラス**」または「**同値パーティション**」と呼びます。
- **なぜ有効か？**
  多くの場合、プログラムは特定の範囲の入力に対して同じロジックで処理を行います。たとえば、「18 歳以上なら成人料金」という仕様の場合、「18 歳」「25 歳」「60 歳」はすべて「成人」という同じグループ（同値クラス）として扱われ、同じ料金計算ロジックが適用される可能性が高いです。そのため、このグループから代表として「25 歳」を選んでテストすれば、他の「18 歳以上」の年齢についても、ある程度の確からしさを確認できる、という考えに基づいています。
- **どのように使うか？（手順の概要）**
  1. テスト対象の入力条件や出力条件を特定します。
  2. 各条件について、「有効な同値クラス」（仕様を満たすデータのグループ）と「無効な同値クラス」（仕様を満たさない、エラーとなるべきデータのグループ）を洗い出します。
     - たとえば、「年齢（0 歳以上 120 歳以下が有効）」という入力条件なら：
       - 無効同値クラス 1: `age < 0` （例: -1）
       - 有効同値クラス: `0 <= age <= 120` （例: 30）
       - 無効同値クラス 2: `age > 120` （例: 130）
     - 数値だけでなく、文字列（空文字、特定の形式）、列挙型なども対象になります。
  3. 各同値クラスから、**代表値を 1 つずつ**選びます。
  4. 選んだ代表値を使って、テストケースを作成します（入力値と期待される結果を定義）。無効同値クラスのテストケースでは、エラー処理が期待通りに行われるかを確認します。
- **例：試験の点数（0 点〜 100 点が有効）に対する同値クラス**
  - 無効同値クラス 1: 点数 < 0 （代表値: -10）→ 期待結果: エラー
  - 有効同値クラス: 0 <= 点数 <= 100 （代表値: 50）→ 期待結果: 正常処理
  - 無効同値クラス 2: 点数 > 100 （代表値: 110）→ 期待結果: エラー
    この場合、最低 3 つのテストケース（-10 点、50 点、110 点）を作成すれば、同値分割法に基づいた基本的なテストが行えます。

同値分割法は、テストケース数を効率的に削減しつつ、入力データの範囲を網羅的にテストするための、もっとも基本的で広く使われるブラックボックステスト技法の 1 つです。

### 3.1.2 境界値分析：境界とその周辺を重点的に狙う

- **どんな考え方か？**
  同値分割法で見つけた「同値クラス」の**境界（端の値）**、およびその**すぐ隣の値**は、プログラムの誤り（バグ）が潜んでいる可能性が高い、という経験則に基づいたテスト技法です。そのため、これらの「境界値」とその周辺を重点的にテストケースとして選択します。
- **なぜ有効か？**
  プログラマは、条件分岐（例: `if (age >= 18)`）やループの終了条件（例: `for (i = 0; i < 10)`）などで、境界値を正しく扱えているか（「以上」と「より大きい」、「未満」と「以下」の区別、ループ回数など）を間違いやすい傾向があります。たとえば、「18 歳以上」のつもりが「18 歳より大きい」と実装してしまったり、配列の最後の要素を処理し忘れたり、といったバグは境界で発生しやすいです。境界値分析は、このような典型的なバグを発見するのに非常に効果的です。
- **どのように使うか？（手順の概要）**
  1. 同値分割法などを用いて、入力条件や出力条件の境界を特定します。
  2. 特定した各境界に対して、以下の値を選択し、テストケースを作成します。
     - **境界値そのもの (On Point):** 境界の値。
     - **境界値のすぐ内側の値 (Off Point - 内側):** 境界値のすぐ隣にある有効な値。
     - **境界値のすぐ外側の値 (Off Point - 外側):** 境界値のすぐ隣にある無効な値。
  3. 選択した値を用いてテストケースを作成し、それぞれの期待される結果（正常処理またはエラー）を確認します。
- **例：試験の点数（0 点〜 100 点が有効）に対する境界値**
  この条件には、「0」と「100」という 2 つの境界があります。
  - **下限境界 (0) の周辺:**
    - -1 （境界の外側・無効）
    - 0 （境界値・有効）
    - 1 （境界の内側・有効）
  - **上限境界 (100) の周辺:**
    - 99 （境界の内側・有効）
    - 100 （境界値・有効）
    - 101 （境界の外側・無効）
      この場合、境界値分析に基づいて、-1, 0, 1, 99, 100, 101 の 6 つの値をテストケースの入力値として選択します。同値分割法で選んだ代表値（例: 50）と合わせることで、より網羅的なテストが可能になります。
- **組み合わせ:**
  境界値分析は、多くの場合、同値分割法と組み合わせて使われます。まず同値分割で大まかなグループ分けを行い、次に各グループの境界付近を境界値分析で重点的にテストする、という流れが一般的です。

境界値分析は、少ないテストケースでバグを発見する確率を高めることができる、非常に費用対効果の高いテスト設計技法です。とくに数値範囲、要素数、日付、時間などを扱う際には、必ず適用を検討すべき技法と言えます。
